## 1. What v0.4.0 is

A serial-release milestone. The browser-local v0.1 Kanban becomes a Supabase-backed single-user planner with a three-mode AI Council (Plan / Advise / Chat), persistent per-user memory, a Council Write Gate that keeps every board change behind a user tap, and a read-only session history. No multi-tenant, no billing, no task branching — those are v0.5 / v0.6 / v1.0. This release ships **one complete voice** and **one complete safety contract**: nothing the Council says can move a card without a tap.

## 3. The release contract (vision §10 + PRD §3.3)

v0.4.0 is the one milestone in this release that is defined as **production-grade**. Per `docs/prd/vision.md` §10 and `docs/releases/v0.4-council/prd.md` §3.3, `v0.4.0` is complete only when **all** of the following are true:

1. All F01–F32 features pass their code + test gates on the release branch.
2. The release branch merges into `main`.
3. The deployment is promoted to **Vercel Production** (not preview).
4. The `v0.4.0` annotated tag exists on `main`.
5. A stable versioned alias (`simple-kanban-v0-4.vercel.app`) is pinned to the production deployment id for permanence.
6. The three acceptance proxies from PRD §3 are walked by the CD (F30 throttled-account, F31 three-naïve-user stopwatch, F32 375px device walk).

Items 1–5 are machine-side and ship in this close-out. Item 6 is CD-side and requires real users on real devices.

```
                         │ \

```

Actual topology — no fast-forward was possible because `main` was one commit (`2406f2a`, the alpha.1 tip) behind the release lineage, so PR #49 landed as a merge commit (`d15f0ec`) whose second parent is the `release/v0.4-alpha` tip (`35433f0`). The `v0.4.0` tag resolves to `cd72275` and is reachable from `main` as an ancestor via that second-parent chain. The tag was originally placed on `release/v0.4-alpha` at `cd72275` on 2026-04-23 (before PR #49 merged), then deleted and re-cut on 2026-04-24 from a `main` branch checkout — at that point `main` HEAD was at the merge commit `d15f0ec`, and the re-cut (`git tag -a v0.4.0 cd72275`) explicitly targeted ancestor commit `cd72275`, not the HEAD commit. The tag continues to point at `cd72275`; the re-cut satisfies PRD §3.3 "tag source: cut from `main` after the release branch merges" literally (the tagging operation was run on a `main` checkout after the merge), and leaves the tag-commit mapping unchanged. Tag object SHA went from `8b8dbd0` (pre-merge) to `6e6ec30` (post-merge, current). `release/v0.4` is the durable long-lived pointer pinned at `cd72275` and never moves. `release/v0.4-alpha` continued to serve as the working branch through the release cycle and now sits at `35433f0` — one commit past `cd72275`, absorbing the docs-only close-out chore from PR #48.

## 6. CD walk-through — before flipping on

Three acceptance proxies from PRD §3 land in the release and are **not** auto-green. Each needs a human — two need real naïve users, one needs a throttled Anthropic account. All three are self-contained runnable documents or unit-tested primitives ready for an end-to-end run.

### 6.1 F30 — Anthropic 429 soft-pause under real throttle

- Mechanism: `lib/council/shared/retryOn429.ts` — exponential backoff with 30s client-side queue cap; shelf enters a "meta-frame" soft-pause indicator on throttle; per-agent failure policy resumes after cap breach.
- Unit coverage: retry primitive + meta-frame protocol + shelf indicator are green in Vitest.
- What's deferred: the PRD §3.3 acceptance proxy specifies verification under a **deliberately throttled tier-1 Anthropic account**. That requires a real account rate-limit configuration, a recorded session artifact, and CD sign-off on user-visible behaviour during the throttle window.
- Verdict rule: one throttled session logged; meta-frame rendered; queue drained within 30s or per-agent failure path reached honestly → GREEN.

### 6.2 F31 — first-run 60-second stopwatch

- Doc: [`docs/releases/v0.4-council/f31-onboarding-qa-protocol.md`](./f31-onboarding-qa-protocol.md)
- What it verifies: three naïve users reach a tappable, trusted proposal in under 60 seconds from first sign-in, per PRD §3 acceptance proxy.
- Evidence the release captures automatically: four client-side `performance.mark` beacons — `council:first-sign-in`, `council:first-greeting-ready`, `council:first-user-submit`, `council:first-proposal-tap`. The DevTools Console snippet in the protocol reads the marks and prints the per-user stopwatch.
- Verdict rule: all three users ≤ 60s → GREEN; any single user > 60s → tighten the path and rerun the whole three-user walk.

### 6.3 F32 — mobile 375px sign-off

- Doc: [`docs/releases/v0.4-council/f32-mobile-375-signoff.md`](./f32-mobile-375-signoff.md)
- What it verifies: every user-tappable element on a real iPhone SE (or Chrome DevTools Device Toolbar at 375 × 667) clears the 44×44 CSS-pixel bar (design-system.md §6.2); the shelf never occludes the active board column header; the morning greeting wraps cleanly.
- Mechanisms the release already enforces: the `tap` token in `tailwind.config.ts` is the single source of truth for 44px; a unit test at `lib/__tests__/tailwind-tokens.test.ts` fails loudly if the token is deleted or retuned; six pre-existing undefined Tailwind utilities (`rounded-radius-sm`, `border-ink-40/70`, `text-ink-50/70/900`-under-old-names, `text-surface-page`) were remapped to canonical §4.2 tokens during the F32 audit.
- Surfaces requiring the walk: `/sign-in`, `/` (empty board + with-tasks + task dialog), shelf composer + ModePicker + chip input + "How I got here" reveal + memory-recall reveal, proposal card approve / try-again, `/history` (search + filters + per-row Delete + pagination), `/settings/council` (A/B/C/D cards + full-purge).

## 7. Known gaps and follow-ups

Not blockers to v0.4.0. Documented so nothing silently rots.

1. **`simple-kanban-v0-4-beta.vercel.app` alias was not set at beta cut.** The `v0.4.0-beta` tag + `release/v0.4-beta` branch preserve the git code at commit `b9ea623`; no Vercel alias was applied. A retroactive alias can still be set by identifying the preview deployment for `b9ea623` and running `npx vercel alias set <dep-url> simple-kanban-v0-4-beta.vercel.app`.
2. **Undefined `--color-ink-300` token used by `/history` and `/settings/council` delete notices.** Pre-existing borderless success-banner styling. A spawned chore (title: *Fix undefined --color-ink-300 token in delete notices*) carries the one-line remap to `border-border-default`. Does not affect tap targets.
3. **Deferred acceptance proxies (F30 throttle verification + F31 stopwatch + F32 375px walkthrough).** Authored and in-repo; require a CD on a real device / real throttled account to execute and append their result to `progress.md` via the templates in each protocol doc. This is the one remaining open front for release GREEN.

## 8. What v0.4.0 does NOT do (scope discipline)

From `CLAUDE.md` project rules — any of these appearing in a v0.4 code path is a bug, not a feature:

- No multi-user / shared Councils / admin tenant-wide toggles. (→ v0.5)
- No multi-list / Tracker view / task branching. (→ v0.6)
- No billing / quotas / public launch. (→ v1.0)
- No agent-runs-code, no auto-apply board changes, no voice I/O, no cross-device sync, no DOM screenshots on error, no public boards or export. (carried across all releases)
